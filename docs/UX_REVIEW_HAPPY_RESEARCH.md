# 使用者体验评审：快乐科研闭环

评审目标：让用户只通过简单对话，就能完成灵感产生、实验设计、数据处理、结果复现、猜想验证、文献互证和论文贡献提炼。

## 当前总体判断

工作流已经具备基础模块，但原先的体验更像“工具箱”。使用者真正想要的是“陪跑式科研闭环”：不知道做什么时能被引导，有数据时能被转成问题，有结果时能被转成证据，有证据时能被提升成论文主张。

本轮已补上中间层：

- `HAPPY_RESEARCH_LOOP.md`: 从灵感到论文贡献的轻量闭环。
- `EXPERIMENT_TO_CLAIM.md`: 从实验结果到论文主张的证据链。
- `05_hypothesis_registry.md`: 把猜想保存成可检验对象。
- `06_result_interpretation.md`: 解释结果支持、反驳还是无法判断猜想。
- `07_claim_evidence_map.md`: 把文献、实验、图表和论文主张连接起来。
- `compare_results.py`: 比较两次 CSV 结果，支持复现检查。

## 用户体验断点与改进

| 断点 | 用户感受 | 已做改进 | 后续还可增强 |
|---|---|---|---|
| 灵感到研究问题之间断开 | “我有想法，但不知道怎么变成研究” | Idea Lab + Socratic 引导 | 自动基于历史日志生成候选方向 |
| 研究问题到实验之间断开 | “知道想研究什么，但不知道怎么验证” | Hypothesis Registry | 自动生成最小可行实验方案 |
| 实验结果到论文主张之间断开 | “有结果，但不知道能不能写进论文” | Result Interpretation + Claim-Evidence Map | 自动生成讨论段落草稿 |
| 复现不够一键化 | “结果跑出来了，但不知道是否可复现” | CSV compare script | 支持图像、模型指标、随机实验容差 |
| 负结果价值不明显 | “不显著是不是失败？” | 文档强调反驳/不确定也有价值 | 增加 negative results log |
| 每天下一步仍需判断 | “今天我该做什么？” | project_status 建议下一步 | 增加 readiness score 和轻任务推荐 |

## 最值得继续改善的 8 件事

1. **数据自动体检**  
   用户上传/指定数据后，自动生成字段类型、缺失值、异常值、样本量、可用分析方向。

2. **最小可行实验设计器**  
   根据假设自动提出最小实验：输入、输出、方法、成功/失败标准、复现方式。

3. **结果卡片自动生成**  
   实验结束后自动生成 result card，写清楚结果、解释、风险和下一步。

4. **负结果日志**  
   把“不支持猜想”的结果保存为边界条件或理论修正材料，而不是丢弃。

5. **主张强度评分**  
   每个 claim 标记 strong / moderate / weak / exploratory，并说明为什么。

6. **项目 readiness score**  
   用 0-100 显示项目成熟度：RQ、文献、假设、实验、结果、图表、写作、完整性。

7. **下一步轻任务推荐**  
   每次结束时推荐一个 15-30 分钟可以完成的小任务，降低继续科研的心理成本。

8. **前沿雷达**  
   对 Idea Lab 中需要 frontier scan 的想法定期做最新文献核查；由于 Codex 不能后台定时运行，这需要在活跃会话中触发。

## 最简单的理想交互

用户说：

```text
我想快乐科研，今天你来带我。先看我们已有积累，然后帮我推进一个最轻松但最有价值的小闭环。
```

Codex 应该：

1. 读取上下文、open loops、Idea Lab、项目状态。
2. 判断当前最适合推进的环节。
3. 只问 1-2 个关键问题。
4. 直接创建或更新对应文件。
5. 输出一个小成果。
6. 归档今天进展，并给出下一步轻任务。
